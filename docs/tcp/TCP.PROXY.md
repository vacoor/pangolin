# TCP 代理后端连接建立策略对比

> 文档版本：v1.0  
> 适用项目：pangolin（TUN 层 TCP 代理）  
> 背景：项目在 TUN 层拦截客户端 TCP 连接，代理转发至后端服务器。

---

## 1. 背景与问题描述

在 TCP 代理场景中，存在一个核心决策点：**代理何时与后端服务器建立连接？** 不同的时机选择会在延迟、资源消耗、安全性、实现复杂度等方面产生显著差异。

本文档深度对比两种策略：

- **策略 A（先连后握）**：收到客户端 SYN 后立即建立后端连接，后端连接成功后才回复客户端 SYN-ACK。这是 pangolin 的**当前实现**。
- **策略 B（先握后连）**：先与客户端完成三次握手，握手完成后再建立后端连接；等待期间用零窗口（`rcvWndFrozen`）冻结客户端发送。

---

## 2. 策略 A：先连后握（当前实现）

### 2.1 工作流程

收到客户端 SYN 时，代理立即发起至后端的 TCP 连接，后端连接结果决定是否向客户端回复 SYN-ACK。

**核心代码路径**（`TcpHandshaker.tcp_conn_request`）：

```
客户端发送 SYN
  → 代理收到 SYN → 创建 tcp_request_sock（状态 TCP_NEW_SYN_RECV）
  → 立即调用 socketChannelFactory.open() 建立后端连接（异步）
  → 后端连接成功 → af_ops.send_synack() 回复客户端 SYN-ACK
  → 客户端回复 ACK → tcp_check_req() 完成握手，进入 ESTABLISHED
  → 后端连接失败 → rsk_ops.send_reset() 向客户端发送 RST
```

### 2.2 时序图

```
客户端 (Client)          代理 (Proxy)             后端 (Backend)
      |                      |                          |
      |------- SYN --------->|                          |
      |                      |-------- CONNECT -------->|
      |                      |       (后端连接中...)     |
      |                      |<------- CONNECTED -------|
      |<----- SYN-ACK -------|                          |
      |------- ACK --------->|                          |
      |                      |                          |
      |====== ESTABLISHED ===|========= ESTABLISHED ====|
      |                      |                          |
      |--- DATA (PSH) ------>|--- DATA (forward) ------>|
```

**关键时间线**：
- `T0`：客户端发出 SYN
- `T1 = T0 + RTT_client`：代理收到 SYN，立即发起后端连接
- `T2 = T1 + RTT_backend`：后端连接完成，代理回复 SYN-ACK
- `T3 = T2 + RTT_client`：客户端收到 SYN-ACK，回复 ACK，握手完成
- **客户端感知到的握手延迟 ≈ RTT_client + RTT_backend + RTT_client**（含一次后端连接 RTT）

### 2.3 状态转换

```
[客户端视角]
SYN_SENT ──(等待 SYN-ACK)──► ESTABLISHED

[代理视角]
LISTEN ──► TCP_NEW_SYN_RECV（等待后端连接）──► 发出 SYN-ACK ──► ESTABLISHED
```

## 3. 策略 B：先握后连（备选方案）

### 3.1 工作流程

代理收到 SYN 后立即回复 SYN-ACK，先与客户端完成三次握手。握手完成（收到客户端 ACK）后，再发起后端连接。等待期间通过通告零接收窗口（`rcvWndFrozen = true`）来阻止客户端发送数据，防止数据在后端连接就绪前到达丢失。

**核心流程**：

```
客户端发送 SYN
  → 代理收到 SYN → 立即回复 SYN-ACK（窗口字段正常）
  → 客户端回复 ACK → 握手完成，进入 ESTABLISHED
  → 代理回复 ACK + 窗口=0（rcvWndFrozen，冻结客户端发送）
  → 代理发起后端连接（异步）
  → 后端连接成功 → 解除零窗口，通告正常接收窗口
  → 客户端感知到窗口打开，开始发送数据
  → 后端连接失败 → 代理发送 RST 或 FIN 终止连接
```

### 3.2 时序图

```
客户端 (Client)          代理 (Proxy)             后端 (Backend)
      |                      |                          |
      |------- SYN --------->|                          |
      |<----- SYN-ACK -------|                          |
      |------- ACK --------->|                          |
      |                      |                          |
      |<== ESTABLISHED ======|                          |
      |                      |-------- CONNECT -------->|
      |<-- ACK(win=0) -------|       (后端连接中...)    |
      |  (零窗口：客户端暂停)  |                          |
      |                      |<------- CONNECTED -------|
      |<-- ACK(win=65535) ---|   (解冻，通告正常窗口)    |
      |                      |========= ESTABLISHED ====|
      |                      |                          |
      |--- DATA (PSH) ------>|--- DATA (forward) ------>|
```

**关键时间线**：
- `T0`：客户端发出 SYN
- `T1 = T0 + RTT_client`：代理收到 SYN，立即回复 SYN-ACK
- `T2 = T1 + RTT_client`：客户端收到 SYN-ACK，握手完成，连接建立
- `T3 = T2`：代理发起后端连接，同时向客户端通告零窗口
- `T4 = T3 + RTT_backend`：后端连接完成，代理解除零窗口限制
- **客户端感知到的握手延迟 ≈ 2 × RTT_client**（不含后端连接 RTT）

### 3.3 零窗口机制说明

零窗口（Zero Window）是 TCP 流量控制的标准机制（RFC 793）。代理在 SYN-ACK 或握手完成后的第一个 ACK 中通告 `rwnd = 0`，客户端因此停止发送数据，进入零窗口探测等待状态。当后端连接就绪后，代理发送 Window Update（`rwnd > 0`），客户端恢复发送。

## 4. 深度对比

### 4.1 连接延迟

| 维度 | 策略 A（先连后握） | 策略 B（先握后连） |
|------|-------------------|-------------------|
| 客户端感知握手延迟 | **2×RTT_client + RTT_backend** | **2×RTT_client**（握手快） |
| 端到端首字节延迟（TTFB） | 2×RTT_client + RTT_backend | 2×RTT_client + RTT_backend（相同） |
| 后端连接时序 | 与客户端握手**并行** | 在客户端握手**之后**串行 |

**分析**：

- 策略 A 中，后端连接与客户端握手完全串行，客户端必须等待后端连接完成才能感知到连接建立。当后端延迟高（如跨区域节点、慢速后端）时，客户端握手时间显著拉长。
- 策略 B 中，客户端握手延迟仅取决于代理自身的响应速度（通常是本地 TUN 处理），握手非常快。但客户端连接建立后需要等待后端连接（被零窗口阻塞），端到端首字节延迟（TTFB）与策略 A 相比并无本质改善，只是"延迟的感受时机"不同。
- 若后端连接速度极快（局域网、同机房），两者差异可忽略不计。

### 4.2 客户端体验

| 维度 | 策略 A（先连后握） | 策略 B（先握后连） |
|------|-------------------|-------------------|
| 连接建立感知 | 慢（含后端 RTT） | 快（仅代理本地延迟） |
| 连接超时风险 | 高（系统默认超时较短，后端慢时触发） | 低（客户端握手快，不易超时） |
| 浏览器 / App 兼容性 | 部分 App 对握手超时敏感，可能提前重试 | 更好，App 感知连接已建立 |
| 数据发送阻塞感知 | 无（握手完成即可发送） | 有（短暂零窗口等待，对用户不可见，但影响吞吐时序） |
| 连接建立失败行为 | 客户端收到 RST，明确感知失败 | 握手后收到 RST/FIN，体验比"握手拒绝"略差 |

**分析**：

- 部分 HTTP 客户端（如浏览器、移动端 SDK）会记录"连接建立时间"（`connect()` 完成时间），策略 B 使这个指标更好看，有利于减少客户端侧的连接超时重试。
- 零窗口期间客户端对用户无感知（数据发送被 TCP 层暂停），但若后端连接迟迟未完成，零窗口持续时间过长会导致客户端零窗口探测频繁，消耗双侧资源。

### 4.3 资源利用

| 维度 | 策略 A（先连后握） | 策略 B（先握后连） |
|------|-------------------|-------------------|
| 无效后端连接开销 | **有**：扫描探测 / 无效请求会触发后端连接 | **较低**：握手完成才连后端，需多一次 RTT 的"过滤窗口" |
| 代理侧半连接资源 | `tcp_request_sock` 持有后端 Channel Future | `TcpSock`（已建立）持有后端 Channel Future |
| 后端连接池压力 | 更高（所有 SYN 都触发后端连接） | 较低（握手失败 / RST 的连接不触发后端建立） |
| 内存占用 | 半连接状态（`TCP_NEW_SYN_RECV`）期间就占用后端资源 | 握手完成后才占用后端资源 |
| SYN Flood 下的后端压力 | **极高**（每个伪造 SYN 都消耗后端连接） | **较低**（SYN 洪水仍消耗代理资源，但不扩散到后端） |

**分析**：

- 策略 A 将流量压力"透传"到后端——任何触达代理的 SYN 都会换来一个后端连接尝试，这在面对扫描或攻击时会显著放大后端压力。
- 策略 B 相当于在代理侧设置了一道"资质过滤"：只有完成握手（证明是真实的 TCP 客户端）才消耗后端连接资源。

### 4.4 错误处理

| 场景 | 策略 A（先连后握） | 策略 B（先握后连） |
|------|-------------------|-------------------|
| 后端连接失败 | 向客户端发 RST（握手未完成，客户端感知连接被拒绝） | 向客户端发 RST 或 FIN（握手已完成，客户端感知连接被中断） |
| 后端连接超时 | 超时期间客户端处于 SYN_SENT，OS 默认超时后重试 | 超时期间客户端处于 ESTABLISHED + 零窗口探测，最终收到 RST |
| 后端断开（已连接后） | 代理向客户端发 RST | 代理向客户端发 RST | 相同 |
| 客户端在等待期发送 RST | 清理 `tcp_request_sock` 和后端连接 Future | 清理 `TcpSock` 和后端连接 Future |
| 错误语义清晰度 | 更清晰（连接建立失败 = 拒绝连接） | 略模糊（连接已建立后被中断 = 连接中断） |

**当前实现的错误处理代码**（`TcpHandshaker.java:143-155`）：

```java
if (future.isSuccess()) {
    af_ops.send_synack(net, parent, req, pkt);
    future.channel().closeFuture().addListener(req.childCloseListener);
} else {
    rsk_ops.send_reset(net, parent, pkt, -88);
    demultiplexer.inet_csk_destroy_sock(req);
}
```

策略 A 的错误处理模型与 Linux 内核的 `tcp_conn_request` 语义高度一致：后端失败等价于"服务不可达"，RST 语义明确。

### 4.5 扫描探测防御

| 维度 | 策略 A（先连后握） | 策略 B（先握后连） |
|------|-------------------|-------------------|
| SYN 扫描（半开扫描）防御 | 扫描者无法完成握手，但代理仍尝试后端连接（**后端暴露**） | 扫描者无法完成握手（SYN-ACK 后不发 ACK），后端不受影响 |
| 端口存活性隐藏 | 无防御：收到 SYN 就回 SYN-ACK，端口明确存活 | 无防御（相同） |
| SYN Flood 抗性 | 弱：每个 SYN 触发一次后端连接 | 较强：后端连接在握手完成后才触发 |
| 代理自身半连接耗尽 | 通过 `sk_acceptq_is_full()` 限制（`maxSynBacklog = 1024`） | 同样需要限制，但后端资源更安全 |
| 探测检测对抗 | 后端会看到来自代理的半连接（可能触发后端安全告警） | 后端只看到完整连接 |

### 4.6 代理检测风险

TCP 代理（特别是 TUN 层）存在被检测识别的风险，常见检测手段：

| 检测维度 | 策略 A 表现 | 策略 B 表现 |
|----------|-------------|-------------|
| 握手延迟异常 | **明显**：握手时间 = 后端 RTT，远超本地延迟，容易被指纹识别 | **较低**：握手时间接近本地响应，更像真实端点 |
| ISN 随机性 | 取决于 `af_ops.init_seq()` 实现（当前基于随机算法） | 相同 |
| TCP 选项一致性 | 代理对客户端协商的 MSS/TS/WSCALE 与对后端协商的不同，存在分裂 | 相同 |
| 零窗口行为 | 无 | **异常**：握手后立即出现零窗口是罕见行为，可能被流量分析器标记 |
| TTL / IP 特征 | TUN 层重写，与真实后端无关 | 相同 |
| 连接建立时间指纹 | 握手时间偏长，统计分析可识别 | 握手时间正常，但零窗口期可能暴露 |

**综合代理检测风险**：
- 策略 A 的主要检测风险在于**握手时间异常**（含后端 RTT）。
- 策略 B 在握手时间上更隐蔽，但引入了**握手后立即零窗口**这一异常特征——合法服务器几乎不会在 `ESTABLISHED` 后立即通告零窗口，这是一个可被流量分析识别的行为特征。

### 4.7 实现复杂度

| 维度 | 策略 A（先连后握） | 策略 B（先握后连） |
|------|-------------------|-------------------|
| 整体复杂度 | 中等 | 较高 |
| 状态机复杂度 | 标准：`TCP_NEW_SYN_RECV → ESTABLISHED` | 增加：需要在 `ESTABLISHED` 后维护"待后端连接"子状态 |
| 零窗口管理 | 不需要 | 需要：正确时机通告零窗口、后端就绪时发送 Window Update |
| 数据缓冲 | 不需要（后端未就绪时客户端无法发送） | 可能需要：零窗口失效窗口期间到达的数据需缓冲 |
| 超时处理 | 一个超时维度（后端连接超时） | 两个超时维度（后端连接超时 + 零窗口持续时间上限） |
| RST 竞态问题 | 存在（`childCloseListener` 与后端连接失败的双重 RST 竞态，代码中已有 `FIXME` 注释） | 相同竞态 + 额外的零窗口解除与 RST 竞态 |
| 与 Linux 内核语义对齐 | 高（与 `tcp_conn_request` 行为一致） | 需要定制，偏离内核标准路径 |
| 与现有代码的兼容性 | 当前实现 | 需改造 `tcp_check_req`、`TcpHandshaker`、`TcpOutput`（零窗口通告逻辑） |

## 6. 适用场景建议

### 推荐使用策略 A（先连后握）的场景

1. **后端响应速度快（< 50ms）**：局域网、同机房后端，后端连接延迟可忽略，策略 B 的收益不明显。
2. **内网代理 / 零信任网关**：客户端均为合法内部用户，无需防御扫描或 SYN Flood，策略 A 的实现简洁性更有价值。
3. **对错误语义有严格要求**：需要让客户端明确区分"连接被拒绝"与"连接被中断"的场景（如 API 网关）。
4. **现有系统迁移成本敏感**：策略 A 是当前实现，维持现状避免引入新 Bug。
5. **后端是可靠的内部服务**：连接失败率极低，后端资源保护需求不强。

### 推荐使用策略 B（先握后连）的场景

1. **后端延迟高（> 200ms）**：跨区域、跨境节点，客户端握手超时风险高，策略 B 显著改善体验。
2. **公网暴露的代理节点**：面对大量扫描探测和 SYN Flood，策略 B 有效隔离后端，降低后端压力。
3. **客户端对握手时间敏感**：移动 App、浏览器等对 `connect()` 延迟有监控告警的场景。
4. **多后端负载均衡**：握手完成后才选择后端，可在握手完成时获取更多上下文（如 TLS SNI、HTTP Host）再做路由决策。
5. **需要隐藏代理特征（握手时间）**：对抗基于握手延迟指纹的检测（但需注意零窗口行为也是特征）。

## 7. 结论

### 当前实现（策略 A）总结

pangolin 当前采用策略 A，与 Linux 内核 `tcp_conn_request` 的语义高度一致，代码结构清晰，与 `TcpHandshaker`、`TcpDemultiplexer`、`TcpInput`（`tcp_check_req`）等核心组件的协作边界明确。其主要不足在于：后端连接延迟被"转移"给了客户端握手，在后端响应慢的场景下客户端体验欠佳；且面对扫描探测时，每一个 SYN 都会触发后端连接尝试，存在将前端攻击放大到后端的风险。

### 策略 B 的取舍

策略 B 在客户端体验和后端保护方面具有优势，但其引入的零窗口机制是一把双刃剑：一方面能阻止数据提前到达，另一方面"握手后立即零窗口"是一个罕见的 TCP 行为，反而在流量特征上引入新的可检测标记。实现上也更为复杂，需要在 `ESTABLISHED` 状态内维护额外的子状态机，增加了竞态条件处理的难度。

### 建议

- **近期**：维持策略 A，针对当前代码中已有的 `FIXME` 注释（双重 RST 竞态、`childCloseListener` 冲突）进行修复，提升现有实现的健壮性。
- **中期**：若 pangolin 需要暴露于公网或面对高延迟后端，可考虑将策略 B 作为可配置选项引入，通过 `SysctlOptions` 风格的配置项控制行为，而非完全替换策略 A。
- **安全加固**：无论采用哪种策略，`maxSynBacklog`（当前为 1024）的限制对于公网场景可能不足，建议配合 SYN Cookie 或更激进的速率限制来应对 SYN Flood。
